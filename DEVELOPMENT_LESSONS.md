# 📝 《海错图》数字化图鉴项目：踩坑与开发经验总结

在《海错图》数字化图鉴项目的迭代与优化过程中，我们针对古籍排版、OCR 解析、图像精确裁切及数据关联等方面遇到了诸多挑战。以下是我们在开发与对齐审计过程中总结的“踩坑点”与解决方案：

---

## 1. ⚠️ PDF 页码系统性偏移与错位
* **踩坑描述**：
  在项目初期，由于古籍 PDF 包含前言、目录及插图的混合排版，导致直接通过页码索引提取图片时出现了**系统性偏移**。例如从第一册第 11 页开始，后续所有的物种配图全部错位了 1~2 页（如飞鱼卡片错显为铜盆鱼和蝘虎鱼的画面）。
* **经验教训**：
  * **切忌盲信自动化提取工具的页码对齐**：古籍 PDF 的逻辑页码（从正文开始）与物理页码（包含封面、序言）往往不一致。
  * **基准点校验机制**：必须在脚本开发前，人工核对并校验前中后三个位置的“标志性物种”（如首页的鲈鱼、中部的石首鱼、尾页的鳄鱼）的物理页码，建立准确的偏移量映射表。

---

## 2. ⚠️ 多物种同框与精细分栏裁切失败
* **踩坑描述**：
  古籍的配图极其不规则，经常出现“一页多图”或“多物种同框”。如果粗暴地采用“一页一图”或“平分二分之一（left/right）”的裁剪逻辑，会导致导出的卡片中同时混入多个物种（例如人鱼卡片里混入了龙鱼的半个身子，鲳鱼卡片显示了邻近鱼的尾巴）。
* **经验教训**：
  * **引入动态分栏算子**：不能使用单一的二分法。在 `rebuild_accurate_database.py` 中引入了 `third_1,2,3`（三分栏）和 `quarter_1,2,3,4`（四分栏）的高精度纵向/横向裁剪机制。
  * **人工审计映射表**：由于自动目标检测（Object Detection）很难完美区分没有边框的古籍手绘插图，最可靠的做法是采用**半自动裁切流水线**——由人工审计确定每页的物种数量及分栏策略，再由 Python 脚本批量无损切片。

---

## 3. ⚠️ 古籍繁简字与异体字的匹配陷阱
* **踩坑描述**：
  在使用 Python 处理 OCR 生成的文本及匹配标题时，古籍原文誊录和 OCR 系统输出大多采用繁体字（如 `鰥魚贊`、`鱽魚`、`贊`），而前端代码和检索正则可能使用的是简体字。这导致正则表达式（如 `[\u4e00-\u9fa5]{1,6}赞`）完全无法匹配，引发数据漏判。
* **经验教训**：
  * **正则与检索的字符归一化**：在处理古籍数据流时，正则表达式必须考虑到繁简异体字的兼容（例如匹配 `[赞贊]`、`[鱼魚]`）。
  * **统一数据源格式**：JSON 数据库中保留高保真的繁体字作为古典艺术呈现，但在检索和系统计算时应统一转化为简体字处理。

---

## 4. ⚠️ 虚拟环境隔离与运行依赖冲突
* **踩坑描述**：
  在执行 `python3 scripts/rebuild_accurate_database.py` 重建数据库时，系统报 `ModuleNotFoundError: No module named 'fitz'`（或缺少 PIL）的错误，但之前明明安装过依赖。这是因为直接使用了系统的全局 Python 解释器，而项目依赖安装在 `.venv` 虚拟环境中。
* **经验教训**：
  * **强制指定 Venv 路径**：在开发指南和脚本运行指引中，一律将执行命令明确写为 `.venv/bin/python3` 或在 README 中引导使用 `bun` 运行。
  * **依赖文档规范化**：及时在 `README.md` 中将所有运行命令（如依赖安装、启动等）向当前项目采用的 `Bun` 运行时及包管理器对齐。

---

## 5. ⚠️ 临时调试图片污染 Git 提交
* **踩坑描述**：
  为了辅助人工审计配图，开发过程中通过脚本生成了数以百计的临时排版检查图片（如 `public/inspect_v1_all/`）。若不小心将其通过 `git add .` 全部 stage 并推送到 GitHub，不仅会极大增加仓库体积，还会干扰生产部署。
* **经验教训**：
  * **规范化 Git 提交检查**：提交前必须执行 `git status`，并在 `git commit` 前用 `rm -rf` 彻底清理临时生成目录。
  * **配置完善的 `.gitignore`**：应将 `inspect_*` 或 `temp_*` 命名的调试目录直接写入 `.gitignore` 中。

---

## 6. ⚠️ “一刀切”的模版式考证导致用户体验空洞
* **踩坑描述**：
  在前端详情页展示的“现代对照与考证”和“食味评价”部分，原先采用了根据分类（如“鱼类”、“介类”）套用固定几句文案的偷懒做法。这导致用户连续翻看多张卡片时发现话术千篇一律（如“海洋鱼类的一种，可能是鲨鱼或比目鱼的近亲”），感觉像是在“糊弄”。
* **经验教训**：
  * **核心词条专属映射 + 兜底生成引擎**：
    * **核心词条**（如河豚、带鱼、比目鱼等）建立**专有对照知识库**，提供高质量的科普信息。
    * **冷门词条**通过 **哈希模板分流** + **古文特征提取**（如自动截取古文中“色银白”、“多细刺”等片段作为上下文），拼装出逻辑合理、独一无二的现代解读，大幅提升了数字化图鉴的趣味性与专业度。
